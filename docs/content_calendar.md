# Content Calendar — Style Superman

這份文件定義內容生產線：**daily brief → 每週選題 → 每月速報 → 主題分析**。它是流程 / 架構文件，不新增排名、銷量、百分比或未驗證趨勢結論。

核心目標：每天抓訊號、每週變成可拍內容、每月回看市場熱度、再把高價值議題沉澱成可長期引用的主題分析。

---

## 1. 內容生產線總覽

```txt
原始來源 / 排行快照 / 社群與媒體訊號
        │
        ▼
Daily Brief（每日）：整理今日 3–5 個 headline trends
        │
        ├──► 每週選題池：挑 3–5 支短影音 / 圖文進排程
        │
        ▼
Monthly Heat Report（每月）：回看歐美品牌 / 單品熱度，標訊號層級與信心
        │
        ▼
Topic Analysis（不定期）：把連續出現、值得壓注的議題寫成深度分析
        │
        ▼
回饋資料層與評分規則：更新來源、taxonomy、權重與下月 watchlist
```

---

## 2. 每日：Daily Brief → 候選選題

**目的**：把當天訊號壓縮成創作者能立即採用的情報。

**輸入**：

- `data/sources.yml` 的來源巡檢結果。
- 新聞 / 品牌公告 / 發售 / 社群觀察等 raw signals。
- `scripts/score_trends.py` 的候選趨勢分數（可選）。
- 既有 taxonomy、brands、people 資料。

**輸出**：

- `reports/daily/YYYY-MM-DD.md`
- 1–3 個可進入本週選題池的候選主題。
- 1–2 個 watchlist 伏筆。

**編輯規則**：

- 寧可少而準，不為湊數把弱訊號寫成 headline。
- 每個 headline 都要能回答：「是什麼 / 為什麼現在 / 觀眾能做什麼」。
- 不確定處標 `待查` 或 `（待驗證）`；不新增沒來源的排名、百分比、銷量。
- 語氣遵守 `docs/content_strategy.md`：自信但不裝、實用優先、快準有態度、誠實。

---

## 3. 每週：選題池 → 發布排程

**目的**：把 daily brief 的情報變成可拍內容，維持穩定發布節奏。

**建議節奏**：每週排 3–5 支，可依人力與平台表現調整。

**週會 / 批次整理清單**：

1. 回看過去 7 天 daily briefs，列出重複出現或分數高的趨勢。
2. 用 `prompts/short_video_ideas.md` 為每個候選趨勢產 2–3 張選題卡。
3. 依內容支柱分配：趨勢科普、實穿教學、觀點 / 爭議、開箱 / 評測、盤點 / 清單。
4. 優先選擇 low / medium 製作難度，避免流程卡在高成本企劃。
5. 對 hot-take 題目做人工審核：有理由、有來源、不嘲笑觀眾、不用假數據製造焦慮。

**輸出**：

- 本週 3–5 支短影音 / 圖文選題。
- 每支選題的 hook、3–5 個段落、CTA、製作難度。
- 下週要追的 watchlist。

---

## 4. 每月：Monthly Heat Report → 市場回看

**目的**：把單日雜訊收斂成月度歐美男裝熱度地圖，供下月選題與主題分析使用。

**輸入**：

- 當月 daily briefs。
- `data/rankings/lyst-index.yml`、`data/rankings/stockx.yml` 的最新季度 / 年度基準。
- 官方發售、聯名、秀程、pop-up、可讀全文媒體報導。
- 電商與社群觀察（若無公開 API 或可回查證據，只能當弱訊號）。

**輸出**：

- `reports/monthly/YYYY-MM-eu.md`
- 本月品牌 / 單品熱度排序（綜合判斷，非官方榜）。
- 訊號來源分層、信心標示、抓取限制。
- 與 Lyst / StockX 基準的差異。
- 2–3 條下月可拍選題或追蹤方向。

**硬規則**：

- 先寫訊號來源分層，再寫 Top list。
- 明確區分 `L2 已確認事件`（聯名、發售、秀程）與 `L4 弱訊號`（電商 best-seller 推測、403 snippet、社群體感）。
- L4 不可支撐「熱賣」「爆紅」「銷量成長」等硬結論；只能寫「待查」「零售端可見」「媒體提及」。
- 若當月訊號弱，報告要誠實降級，不硬湊 Top 10。

## 5. 不定期：Topic Analysis → 長線資產

**目的**：把連續出現、值得壓注的題目沉澱為可長期引用的分析，不讓情報只停在日報。

**觸發條件**（符合任一即可）：

- 同一趨勢連續 2–4 週出現在 daily / monthly 產物。
- 同時有 L1/L2 強訊號與內容表現數據支持。
- 對 Style Superman 的受眾有明確實穿價值或爭議價值。
- 需要釐清市場誤解，例如「媒體在講，但一般男性真的能穿嗎？」

**輸出位置**：`reports/analysis/`

**分析重點**：

- 趨勢定義與來源。
- 男裝視角的實穿 / 消費 / 文化解讀。
- 反方觀點與風險。
- 可以延伸成哪些短影音、圖文或導購題。
- 後續要回看的指標。

---

## 6. 回饋迴路

每次完成週選題、月報或主題分析，都要回頭檢查：

- `data/sources.yml`：是否有新來源值得加入？舊來源是否失效？
- `data/trend_taxonomy.yml`：是否有分類裝不下的新現象？
- `docs/trend_scoring_rules.md`：評分權重是否符合實際命中率？
- `prompts/`：是否出現固定弱點，需要像月報一樣回頭硬化提示詞？
- `reports/`：是否留下可回看的證據鏈，而不是只留結論？

## 7. 角色分工

| 階段 | Codex | Claude Code | 人類 |
|------|-------|-------------|------|
| Daily brief 判斷 | 主編判斷 / prompt 調校 | 產骨架 / 工程支援 | 最終採用 |
| 每週選題 | 選題架構 / 語氣檢查 | 批次整理可輔助 | 決定拍攝 |
| Monthly report | 訊號分層 / 信心審核 | 自動化支援 | 品牌判斷 |
| Topic analysis | 架構與內容整合 | 格式 / tooling 輔助 | 發布決策 |

同一輪 PR 只做同一主題。內容層變更不要混入工程腳本；工程自動化也不要未經確認改寫內容事實。
